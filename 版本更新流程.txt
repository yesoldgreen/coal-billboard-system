================================================================================
一、版本更新标准流程
================================================================================

每次版本更新时，必须按照以下流程执行：

【第一步：代码修改】
1. 修改客户端代码（如需要）
2. 修改后端代码（如需要）
3. 修改数据库结构（如需要）
4. 更新版本号

【第二步：文档更新】
1. 更新【更新记录.txt】文件
   - 在文档开头添加新版本的完整更新内容
   - 包含：版本号、日期、新增功能、技术实现、使用说明等
   - 保持详细和完整

2. Session中输出简洁版更新日志
   - 只输出核心功能点和使用提示
   - 突出最重要的变化
   - 提供访问地址

【第三步：测试部署】
1. 重启服务
2. 导出端口
3. 验证功能

================================================================================
二、文档输出规范
================================================================================

【在对话Session中输出 - 简洁版】

格式：
```
## ✅ vX.X.X 版本升级完成！

### 📋 更新内容
- 核心功能点1
- 核心功能点2

### 🌐 访问地址
- 客户端：[URL]
- 管理后台：[URL]

### 💡 使用提示
1. 简要说明如何使用新功能
2. 注意事项（如有）
```

特点：
- 简洁明了
- 只保留核心信息
- 便于用户快速理解

--------------------------------------------------------------------------------

【在更新记录.txt中输出 - 完整版】

格式：
```
【vX.X.X - 日期】

新增功能：
- 详细的功能描述
  * 子功能1
  * 子功能2

技术实现：
- 数据库修改
- 后端修改
- 前端修改

使用说明：
1. 详细的操作步骤
2. 注意事项
3. 示例说明

视觉效果/技术细节：
- 具体的实现细节
- 代码示例
- 配置说明
```

特点：
- 详细完整
- 包含所有技术细节
- 便于后续查阅和维护

================================================================================
三、版本号命名规范
================================================================================

格式：vX.Y.Z

- X：大版本号（重大功能更新或架构变化）
- Y：小版本号（新功能添加）
- Z：修订号（Bug修复、小优化）

示例：
- v1.4.6  → 新增箭头符号功能
- v1.4.5.1 → 修复颜色显示bug
- v1.5.0  → 重大功能更新（如果有）

================================================================================
四、更新记录.txt维护规范
================================================================================

【文档结构】
1. 版本更新历史（按时间倒序，最新版本在最前）
2. 系统概述
3. 功能介绍
4. 使用指南
5. 技术说明
6. 常见问题

【每次更新时】
- 在"版本更新历史"部分的最开头添加新版本内容
- 更新文档头部的"当前版本"和"最后更新"时间
- 如果新功能影响到功能介绍或使用指南，同步更新对应部分

【保持一致性】
- 每个版本使用相同的格式
- 使用分隔线（--------）区分不同版本
- 保持术语统一

================================================================================
五、检查清单
================================================================================

每次版本更新完成后，确认以下事项：

代码层面：
□ 代码已修改并测试
□ 版本号已更新（客户端标题）
□ 数据库迁移脚本已创建并执行（如需要）

文档层面：
□ 更新记录.txt已更新（完整版）
□ Session中已输出简洁版更新日志
□ 新版本内容已添加到文档开头
□ 文档头部版本号和日期已更新

部署层面：
□ 服务已重启
□ 端口已导出
□ 访问地址已提供给用户
□ 功能已验证

六、示例对比
================================================================================

【Session输出 - 简洁版示例】

## ✅ v1.4.6 版本升级完成！

### 📋 更新内容
- 排队中数据添加箭头符号（↓ 减少/绿色，↑ 增加/红色）
- 文档重命名为【更新记录.txt】

### 🌐 访问地址
- 客户端：https://xxx/client/index.html?id=1
- 管理后台：https://xxx/admin/

### 💡 使用提示
更新排队数据后，排队中数字会显示颜色和箭头，直观显示变化趋势。

--------------------------------------------------------------------------------

【更新记录.txt - 完整版示例】

【v1.4.6 - 2026年3月10日】

新增功能：
- 排队拉运模块"排队中"参数添加箭头符号
  * 数字减少时：显示绿色文字 + 向下箭头 ↓
  * 数字增加时：显示红色文字 + 向上箭头 ↑
  * 更直观地表达排队数量的变化趋势

技术实现：
- 在客户端渲染逻辑中添加箭头符号
- 使用Unicode字符 ↓ (U+2193) 和 ↑ (U+2191)
- 箭头颜色与数字颜色保持一致

视觉效果：
- 133 → 130  显示为：130 ↓ （绿色）
- 133 → 140  显示为：140 ↑ （红色）
- 133 → 133  显示为：133    （黑色，无箭头）

使用说明：
1. 在后台更新排队拉运数据
2. 客户端会自动比较与上次的差异
3. 减少显示绿色+向下箭头，增加显示红色+向上箭头
4. 一目了然地看出排队趋势

文档更新：
- 将【使用指南.txt】重命名为【更新记录.txt】
- 每次版本更新时自动添加更新内容到文档开头

================================================================================
七、注意事项
================================================================================

1. 保持简洁原则
   - Session输出要简洁，让用户快速了解核心变化
   - 详细内容放在文档中，供深入查阅

2. 保持一致性
   - 每次更新使用相同的格式
   - 术语保持统一

3. 用户视角
   - 从用户角度描述功能（而不是技术实现）
   - 提供具体的使用场景和示例

4. 及时更新
   - 每次版本更新都要同步更新文档
   - 不要积累多个版本再一起更新

5. 保持历史
   - 不要删除旧版本的更新记录
   - 按时间倒序排列，方便查看演进历史
